Part 1 - Service Architecture and Setup
Question: Explain the default lifecycle of a JAX-RS Resource class. Is a new instance created per request or is it a singleton? How does this affect your in-memory data management?
By default, JAX-RS creates a new instance of a resource class for every incoming request. This means you cannot store any shared state inside the resource class itself because each request gets its own fresh object. To handle this, I kept all shared data inside the MockDatabase class using static ConcurrentHashMap fields. Because Grizzly processes requests on multiple threads, a regular HashMap would risk data corruption under concurrent access, so ConcurrentHashMap was the right choice to keep things thread-safe without any extra synchronization code.
Question: Why is HATEOAS considered a hallmark of advanced REST design and how does it benefit client developers?
HATEOAS means the API tells clients where they can go next by including links inside responses, rather than forcing them to read separate documentation and hardcode URLs. For example, the discovery endpoint returns links to /api/v1/rooms and /api/v1/sensors so a client can find everything from that one starting point. This makes the API easier to explore and reduces the chance of clients breaking when paths change, because they follow links rather than relying on URLs they memorized.
Part 2 - Room Management
Question: When returning a list of rooms, what are the implications of returning only IDs versus returning full room objects?
Returning only IDs is lighter on bandwidth but forces the client to make a separate GET request for every room it needs to display. Returning full objects uses more bandwidth but saves the client from doing multiple round trips. For this API I return full room objects because the Room model is small and the benefit of having everything in one response outweighs the slightly larger payload.
Question: Is the DELETE operation idempotent in your implementation? What happens if the same DELETE request is sent multiple times?
The first DELETE on a room that exists and has no sensors returns 204 and removes the room. If the exact same request is sent again, the room is already gone so the server returns 404. Strictly speaking this means the response changes between calls, but the server state does not change after the first successful deletion. In REST, idempotency refers to the effect on server state rather than the response code, so DELETE is still considered idempotent in this implementation.
Part 3 - Sensor Operations and Linking
Question: What happens if a client sends data in a format other than JSON when @Consumes(MediaType.APPLICATION_JSON) is declared?
JAX-RS checks the Content-Type header of the incoming request against what the method declares it can consume. If a client sends text/plain or application/xml instead of application/json, the framework rejects the request before it even reaches the method and returns a 415 Unsupported Media Type response. No application code runs at all, which is a clean way to enforce the contract without writing any manual validation.
Question: Why is using @QueryParam for filtering considered better than putting the filter value in the URL path?
A path like /sensors/type/CO2 implies that CO2 is a distinct resource with its own identity, which is misleading. A query parameter like /sensors?type=CO2 clearly signals that this is a filter applied to the sensors collection, not a resource itself. Query parameters are also optional by nature, so the same endpoint handles both the full list and the filtered list without needing two separate methods. This is the standard REST convention for search and filtering.
Part 4 - Sub-Resources
Question: What are the architectural benefits of the Sub-Resource Locator pattern?
Rather than cramming every possible path into one large resource class, the sub-resource locator lets you delegate responsibility to a dedicated class. SensorResource handles
/Sensors and hands off to SensorReadingResource for anything under
/sensors/{id}/readings. This keeps each class focused on one thing, makes the code easier to read, and means changes to reading logic do not touch sensor logic at all. As an API grows this separation becomes essential because a single controller class with dozens of methods quickly becomes unmanageable.
Part 5 - Error Handling and Logging
Question: Why is HTTP 422 more semantically accurate than 404 when a referenced resource is missing inside a valid JSON payload?
A 404 means the URI you requested was not found, which would confuse a client because the URI /api/v1/sensors is perfectly valid. The problem is not the address but the content of the request body, specifically the roomId value pointing to something that does not exist. 422 means the server understood the request and the JSON is valid, but it cannot process it because of a logical problem inside the data. That is a much more honest description of what went wrong.
Question: From a cybersecurity standpoint, what are the risks of exposing Java stack traces to external API consumers?
A stack trace reveals the internal structure of the application, including class names, method names, line numbers, and sometimes library versions. An attacker can use this to identify which frameworks are in use and look up known vulnerabilities for those specific versions. Method names and class paths can also hint at the overall architecture and help an attacker plan a more targeted exploit. The catch-all ExceptionMapper prevents any of this from leaking by replacing the raw trace with a generic 500 response that gives nothing useful away.
Question: Why is it better to use JAX-RS filters for logging rather than putting Logger.info() calls inside every resource method?
Putting logging calls inside every method means the same boilerplate code is repeated across the entire codebase. If the log format ever needs to change, every single method has to be updated. A filter is a cross-cutting concern, meaning it applies to every request and response automatically without touching any resource class. This keeps the resource methods focused purely on business logic and makes the logging behavior easy to change or extend in one place.
